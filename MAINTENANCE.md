# GameList — заметки для сопровождения

Актуально на 2026-07-17. Это техническая карта проекта; пользовательское описание и запуск находятся в `README.md`, исходный план — в `PLAN.md`, остаток задач — в `TODO.md`.

## 1. Назначение и основные правила

GameList — русскоязычное Laravel-приложение для личных и публичных списков игр. Основные инварианты:

- публичный список доступен только для чтения по `/{login}/{slug}`;
- публичный профиль доступен по `/{login}`;
- собственная страница есть только у игры из `catalog_games`: `/game/{catalog_games.id}`;
- ручная игра остаётся без отдельной страницы (`games.catalog_game_id = null`);
- внешние изображения всегда скачиваются, оптимизируются в WebP и хранятся локально; в `catalog_games` хранится только исходный URL;
- интерфейс русский, строки приложения находятся в `lang/ru`; новые тексты не следует без необходимости зашивать в PHP;
- production-ассеты `public/build` хранятся в Git, потому что на хостинге нет Node.js.

## 2. Стек и инфраструктура

- PHP 8.3+, Laravel 13, MySQL, Blade.
- Tailwind CSS 4 + Vite 8, основной CSS — `resources/css/app.css`, JS — `resources/js/app.js`.
- HowLongToBeat: `aneeskhan47/php-howlongtobeat-api`, скрыта за `App\Contracts\GameCatalog`.
- Почта: `resend/resend-php`, настройки только через окружение.
- Изображения: PHP GD, WebP, диск Laravel `public` и симлинк `public/storage`.
- Иконки: локальный поднабор Google Material Symbols в `public/fonts/material-symbols-outlined.woff2`.
- Сессии, cache и queue настроены на БД. Текущие письма отправляются через `defer`, отдельный queue worker для них не требуется.

Ключевые переменные окружения перечислены в `.env.example`. Секреты, API-ключи и production `.env` не коммитить.

## 3. Модель данных

| Таблица / модель | Назначение |
|---|---|
| `users` / `User` | login, необязательный уникальный email, пароль, аватар, обложка профиля, активность |
| `game_lists` / `GameList` | список, slug, публичность, обложка, платформа, доступные статусы, режим отображения |
| `games` / `Game` | игра в пользовательском списке, статус, даты, локальная обложка, HLTB-метрики, необязательная связь с каталогом |
| `catalog_games` / `CatalogGame` | локальный кэш HLTB: название, URL обложки, сюжет и 100% |
| `game_reviews` / `GameReview` | одна оценка 1–10 и одно Markdown-мнение пользователя на каталожную игру |
| `friendships` | направленная связь `user_id -> friend_id`, подтверждение не требуется |
| `favorite_games` | до трёх любимых игр пользователя с порядком |
| `notifications` | стандартные постоянные database notifications Laravel; удаляются вручную |
| `password_reset_otps` | хеш шестизначного OTP, срок действия и число попыток |

Схема создаётся миграциями из `database/migrations`. При изменении существующей схемы добавлять новую миграцию, старые production-миграции не переписывать.

## 4. Статусы, платформы и даты

Источники истины:

- `App\Enums\GameStatus`: `want_to_play`, `installed`, `playing`, `completed`, `dropped`, подпись и Material icon;
- `App\Enums\Platform`: Nintendo Switch, Steam, PC, PlayStation, Xbox;
- переводы — `lang/ru/app.php`.

У списка можно включить от одного до пяти статусов (`game_lists.available_statuses`). Отключить уже используемый статус нельзя. Первый включённый статус становится статусом по умолчанию.

`Game::saving()` автоматически устанавливает:

- `started_at = today()` при переходе в «Играю», если дата пуста;
- `completed_at = today()` при переходе в «Пройдена», если дата пуста.

Даты при последующих сменах статуса не очищаются. История строит отдельные события начала и завершения и сортирует новые события сверху. Продолжительность прохождения выводится в днях/месяцах через `Game::completionDuration()`.

## 5. Списки и игры

Главные контроллеры: `GameListController`, `PublicListController`, `GameController`, `GameImportController`.

- Режимы списка: `cards`, `compact`, `board` («Доска»).
- Фильтр принимает несколько `status[]` и сохраняется при смене сортировки.
- Сортировки: `added_at` (порядок списка + новые сверху) и `completed_at` (последние завершённые сверху, пустые даты внизу).
- Кнопка копирования формирует текущую отфильтрованную выдачу как `- Название игры`.
- Импорт принимает до 100 игр, одна непустая строка — одна игра. `GameImportParser` очищает маркеры `-`, `*`, `+`, чекбоксы Markdown и числовую нумерацию, затем показывает дубликаты.
- Дубликат внутри одного списка определяется по `normalized_title`; нормализация находится в `GameTitleNormalizer`.
- Доступ к изменению списка/игры проверяется по владельцу в контроллерах.

Обложка списка используется как фон в листинге и полном просмотре. При загрузке новой обложки старая удаляется только после успешной записи новой; отдельной команды «удалить текущую» нет.

### AJAX-смена статуса

Маршрут: `PATCH /games/{game}/status`. Форма отправляется JS как `POST` с `_method=PATCH`, CSRF и `Accept: application/json`. Сервер возвращает новый статус, иконку и даты.

`resources/js/app.js` без перезагрузки:

- обновляет карточку и автоматически выставленные даты;
- перемещает игру между колонками доски;
- пересчитывает колонки и содержимое кнопки копирования;
- удаляет игру из текущей выдачи, если новый статус не проходит активный фильтр;
- сохраняет выбранную сортировку DOM.

Критически важно: `FormData` нужно создавать **до** `select.disabled = true` и явно делать `formData.set('status', select.value)`. Disabled-поле не включается браузером в форму; нарушение этого правила уже приводило к ответу 302/валидации и возврату старого статуса.

## 6. Каталог и HowLongToBeat

Поток поиска двухфазный:

1. `CatalogGameCache` сразу ищет в MySQL `catalog_games`.
2. JS вызывает `CatalogSearchController::fresh()`, `HowLongToBeatCatalog` получает свежие данные, кэш обновляется через upsert, затем каталог перерисовывается.

Основные страницы и ограничения:

- `/search` доступен всем; для гостя кнопка `+` ведёт ко входу;
- `/lists/{list}/catalog` сразу привязан к конкретному списку;
- выдача ограничена 20 элементами на страницу, продолжение загружается без перезагрузки;
- быстрое добавление возвращает `201`, дубликат — `409`;
- если HLTB недоступен, локальные результаты остаются, а игру можно добавить вручную.

`CatalogGameListAdder` — единая точка добавления каталожной игры в список. Он проверяет дубликат по `catalog_game_id`, `normalized_title` и `hltb_id`, скачивает/копирует локальную обложку и сохраняет метрики. `GameObserver::saving()` связывает пользовательскую игру с `catalog_games` по `hltb_id`.

Чтобы заменить HLTB, реализовать `GameCatalog::search()` и поменять binding в `AppServiceProvider`; контроллеры от конкретной библиотеки не зависят.

## 7. Изображения

Вся обработка находится в `CoverImageService`:

- максимум 8 МБ;
- game cover: до 900×1200;
- list cover: до 1800×1200;
- profile cover: до 1800×700;
- avatar: до 512×512;
- WebP quality 82;
- при замене старый файл удаляется;
- URL разрешены только `http/https`, DNS должен указывать на публичный IP; редиректы проверяются заново, максимум четыре.

Каталожный `cover_url` — внешний адрес для кэша/предпросмотра. После добавления игры в список фактическая обложка должна быть в `storage/app/public/...`.

## 8. Страницы игр и отзывы

`GamePageController` обслуживает `/game/{catalogGame}`:

- общие добавления по статусам (заголовок блока скрыт, сами счётчики сохранены);
- средняя оценка и число оценок;
- добавление в один из собственных списков с выбором доступного статуса;
- собственная оценка 1–10 и Markdown-мнение;
- AJAX-предпросмотр Markdown;
- публичные мнения по 10 на страницу.

Markdown проходит через `ReviewMarkdown`: HTML удаляется, unsafe links запрещены. Один пользователь имеет не более одного `game_reviews` на игру.

Ссылки на детальную страницу выводятся из каталога, списков, главной, любимых игр и профильных колонок только при ненулевом `catalog_game_id`.

## 9. Профили, друзья и активность

Компонент плашки — `resources/views/components/profile-card.blade.php`, статистика — `UserProfileStats`.

- Золотая рамка аватара означает наличие email, серебряная — его отсутствие.
- При отсутствии аватара используется Material icon `person`.
- Публичная статистика и три колонки «Хочу сыграть / Играю / Пройдена» считают только игры из публичных списков.
- В колонках выводятся максимум три игры: по `created_at`, `started_at`, `completed_at` соответственно.
- Любимые игры выбираются только из собственных списков через autocomplete, максимум три; они видны публично.
- Дружба направленная. «Мои друзья» — добавленные текущим пользователем; «Со мной хотят дружить» — добавившие его без взаимной связи.

`TrackUserActivity` подключён ко всему web middleware stack и обновляет `last_seen_at` на каждом запросе авторизованного пользователя, включая refresh публичной страницы. `User::isOnline()` считает пользователя онлайн менее часа; затем `activityLabel()` выводит часы, дни или месяцы.

## 10. Уведомления и почта

Observers регистрируются в `AppServiceProvider`:

- `GameListObserver`: новый/впервые опубликованный список;
- `GameObserver`: новая игра в публичном списке, начало и завершение игры;
- `FriendController`: добавление в друзья.

Уведомления получают пользователи, которые добавили автора события в друзья (`followers` автора). Сам владелец получает пожелание удачи при статусе «Играю» и поздравление при «Пройдена». Панель уведомлений непрозрачная, записи хранятся до удаления; удаление и очистка работают по AJAX с обычной отправкой как fallback.

`UserMailService` изолирует отправку и логирует ошибки без падения пользовательского запроса. Письма:

- добавление в друзья;
- OTP восстановления;
- подтверждение смены пароля;
- напоминание после месяца отсутствия.

OTP: 6 цифр, 10 минут, максимум 5 проверок; выдача ограничена 3 запросами в минуту на email. При успешном сбросе удаляются остальные сессии пользователя.

Фоновые задачи без cron:

- после обычного веб-запроса, но не чаще раза в сутки — письма неактивным пользователям;
- при выдаче нового OTP — удаление всех просроченных OTP.

Cron на production не требуется. Нужны `MAIL_MAILER=resend`, `RESEND_API_KEY`, корректные `MAIL_FROM_ADDRESS` и `APP_URL`; ключ хранить только в окружении.

## 11. Frontend и Material Symbols

Blade-шаблоны находятся в `resources/views`, общая навигация — `layouts/app.blade.php`. JS написан без фреймворка через делегирование событий и `data-*` hooks. Основные AJAX-сценарии: поиск каталога, lazy load, быстрое добавление, уведомления, Markdown preview и смена статуса.

Визуальные договорённости:

- тёмная адаптивная тема с glass-панелями; проверять desktop и mobile;
- фоновые обложки списков и профиля затемняются градиентом, но изображение должно оставаться различимым;
- карточки списков сохраняют пустой `<p>` под описание для одинаковой высоты, но не выводят fallback «Игровая коллекция без описания»;
- главная показывает до трёх случайных игр с обложками из `catalog_games`, подписи `@zerocool`, `Steam` и статусы «Хочу сыграть / Играю / Пройдена»; каталожные карточки ведут на страницу игры;
- навигация авторизованного пользователя содержит списки, историю, друзей, профиль, настройки, поиск, уведомления и выход;
- favicon, PNG-иконки, Apple Touch Icon и `site.webmanifest` лежат в `public`.

Material Symbols self-hosted, чтобы при загрузке не показывались текстовые ligature names. В layout используются preload, `font-display: block` и версия URL для cache busting; CSS фиксирует ширину/overflow иконки.

Шрифт — оптимизированный поднабор. При добавлении нового названия иконки:

1. собрать **алфавитный** список всех используемых ligature names, включая динамические имена из enum/JS;
2. запросить Google Fonts CSS по шаблону `https://fonts.googleapis.com/css2?family=Material+Symbols+Outlined&icon_names=<comma-list>&display=block`;
3. скачать указанный WOFF2 в `public/fonts/material-symbols-outlined.woff2`;
4. увеличить `?v=...` одновременно в preload и `@font-face` в `layouts/app.blade.php`;
5. проверить, что название формируется в один glyph, а не выводится текстом.

## 12. Локализация и расширение

Текущий locale — `ru`, fallback — `en`. Для английского создать эквивалентные файлы `lang/en/*`; enum уже берут подписи через `__()`.

Типовые расширения:

- новая платформа: добавить case в `Platform`, перевод и тест; поля БД строковые;
- новый статус: добавить case/иконку в `GameStatus`, переводы `statuses` и `profile_statuses`, проверить формы, статистику, доску, уведомления и шрифт;
- новый источник каталога: реализовать `GameCatalog`, сохранить контракт результата и заменить binding;
- новый тип изображения: использовать `CoverImageService`, не писать отдельное скачивание без SSRF-проверки;
- новый публичный блок профиля: обязательно ограничить запрос публичными списками, если данные не должны раскрывать приватные коллекции;
- новый AJAX endpoint: оставить обычный non-JS fallback, для JSON использовать `Accept: application/json` и покрыть Feature-тестом.

## 13. Проверка и деплой

Перед завершением изменения:

```bash
./vendor/bin/pint --dirty
php artisan view:cache
php artisan test
npm run build
git diff --check
```

Если UI/CSS/JS не менялись, Vite-сборку можно не обновлять. Если менялись — новые хешированные файлы и `public/build/manifest.json` должны попасть в Git, а старые хешированные файлы удалиться.

Production checklist:

```bash
composer install --no-dev --optimize-autoloader
php artisan migrate --force
php artisan storage:link
php artisan config:cache
php artisan route:cache
php artisan view:cache
```

Также проверить права на `storage` и `bootstrap/cache`, MySQL, PHP GD, Resend и соответствие `APP_URL` публичному домену. На сервере не запускать npm; использовать уже собранный `public/build`.
